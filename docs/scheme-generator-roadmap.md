# Конфигуратор принципиальных схем теплового пункта

> **Ветка:** `feature/scheme-generator`
> **Дата:** 2026-04-20
> **Статус:** В работе

---

## 1. Проблема

Текущий воркфлоу: сервис отправляет клиенту PDF-образцы схем (принципиальная схема, план теплового пункта, план подключения) и просит клиента самостоятельно изобразить схемы и загрузить на upload-страницу. Это создаёт барьер — большинство клиентов не могут нарисовать инженерную схему и затягивают процесс.

## 2. Решение

Заменить загрузку PDF-схем на **интерактивный конфигуратор**: клиент отвечает на вопросы (радиокнопки/чекбоксы), система подбирает типовую схему из библиотеки SVG-шаблонов, подставляет параметры из `parsed_params` и генерирует готовый PDF с ГОСТ-рамкой.

### Scope MVP

MVP — только **принципиальная схема теплового пункта** (`heat_scheme`).
План расположения УУТЭ и план подключения — следующие этапы.

## 3. Каталог конфигураций

8 типовых схем, определяемых деревом решений из 4 параметров:

| № | Тип присоединения | Клапан / насос | ГВС (2-ступ. подогреватель) | Вентиляция (параллельная) |
|---|-------------------|----------------|----------------------------|--------------------------|
| 1 | Зависимая | Без клапана | Нет | Нет |
| 2 | Зависимая | Без клапана | Да | Нет |
| 3 | Зависимая | 3-ходовой + насос на перемычке | Нет | Нет |
| 4 | Зависимая | 3-ходовой + насос на перемычке | Да | Нет |
| 5 | Зависимая | 3-ходовой + насос на перемычке | Да | Да |
| 6 | Независимая | 2-ходовой + насос + G3 подпитка | Нет | Нет |
| 7 | Независимая | 2-ходовой + насос + G3 подпитка | Да | Нет |
| 8 | Независимая | 2-ходовой + насос + G3 подпитка | Да | Да |

### Дерево решений (вопросы клиенту)

```
Q1: Тип присоединения
    ├─ Зависимая
    │   └─ Q2: Есть ли трёхходовой клапан с насосом на перемычке?
    │       ├─ Нет  → Q3
    │       └─ Да   → Q3
    ├─ Независимая  → Q3
    └─ Не знаю      → scheme_config.connection_type = null (определит инженер)

Q3: Есть ли система ГВС с двухступенчатым подогревателем?
    ├─ Нет  → Q4 (пропуск, вентиляция невозможна без ГВС в текущем каталоге)
    ├─ Да   → Q4
    └─ Не знаю → scheme_config.has_gwp = null

Q4: Есть ли параллельное включение вентиляции?
    ├─ Нет  → Готово
    ├─ Да   → Готово
    └─ Не знаю → scheme_config.has_ventilation = null
```

### Автозаполнение из parsed_params

Если ТУ уже распарсены, система предзаполняет ответы:
- `parsed_params.connection.connection_type` → Q1
- `parsed_params.connection.system_type` → подсказка для Q2
- `parsed_params.connection.heating_system` → определение наличия ГВС/вентиляции

Клиент видит предзаполненные значения и может подтвердить или изменить.

## 4. Архитектура

### Пайплайн данных

```
Клиент (upload.html)
    │
    ▼
Ответы на вопросы (SchemeConfig JSON)
    │
    ▼
POST /api/v1/schemes/preview
    │ → resolve_scheme(config) → имя SVG-шаблона
    │ → render_scheme_svg(template, params) → SVG строка
    │
    ▼
Превью в браузере (inline SVG)
    │
    ▼ [Клиент нажимает «Подтвердить»]
    │
POST /api/v1/schemes/{order_id}/generate
    │ → SVG + ГОСТ HTML-обёртка → WeasyPrint → PDF bytes
    │ → сохранение OrderFile(category=heat_scheme)
    │ → scheme_config → survey_data.scheme_config
    │
    ▼
PDF с ГОСТ-рамкой и штампом в файлах заявки
```

### Подход к SVG: программная генерация (svgwrite)

Каждая схема генерируется Python-кодом через `svgwrite`. Преимущества:
- Легко итерировать и менять элементы
- Параметры подставляются программно (не Jinja2-плейсхолдеры)
- Проще добавлять новые конфигурации
- Единый стиль через константы

### Элементы SVG (инженерные символы ГОСТ)

Каждый элемент — функция, возвращающая SVG-группу:

| Элемент | Символ | Описание |
|---------|--------|----------|
| `draw_pipe(...)` | Линия (толстая) | Трубопровод (подача / обратка) |
| `draw_gate_valve(...)` | ⊠ | Задвижка |
| `draw_filter(...)` | ▽ | Фильтр-грязевик |
| `draw_flowmeter(...)` | ⊕ G1/G2/G3 | Расходомер |
| `draw_temp_sensor(...)` | ● | Датчик температуры |
| `draw_pressure_sensor(...)` | ○ | Датчик давления |
| `draw_calculator(...)` | Прямоугольник УУТЭ | Тепловычислитель с таблицей |
| `draw_pump(...)` | ⊂⊃ | Насос |
| `draw_valve_3way(...)` | ◁ | Трёхходовой клапан |
| `draw_valve_2way(...)` | ◁ | Двухходовой клапан |
| `draw_heat_exchanger(...)` | ≋ | Теплообменник (подогреватель) |
| `draw_radiator(...)` | ▤ | Радиатор (отопление) |
| `draw_check_valve(...)` | ▷ | Обратный клапан |
| `draw_gost_frame(...)` | Рамка + штамп | ГОСТ рамка A3/A4 |

### PDF-генерация

Стек: SVG → HTML (Jinja2 обёртка с ГОСТ-рамкой) → WeasyPrint → PDF.

WeasyPrint выбран потому что:
- Чистый Python, не нужен headless-браузер (Puppeteer)
- Хорошо работает с SVG
- Точное управление размерами страниц (A3, A4)
- Работает в Docker без дополнительных зависимостей (кроме системных библиотек)

## 8. Технические решения и ограничения

### Выбор подхода: программная генерация SVG (Путь Б)

Каждая схема генерируется Python-кодом (`svgwrite`), а не рисуется вручную в Inkscape.

**Плюсы:**
- Быстрее для итераций и правок
- Параметры подставляются программно
- Проще добавлять новые конфигурации
- Единый стиль через константы
- Версионирование в Git

**Минусы:**
- Первоначальная разработка сложнее
- Нужна точная геометрия элементов

### WeasyPrint vs Puppeteer

WeasyPrint выбран, т.к. Puppeteer не установлен на сервере. WeasyPrint — чистый Python, работает в Docker.

### Ограничения MVP

- Только принципиальная схема теплового пункта (heat_scheme)
- 8 фиксированных конфигураций (не модульная сборка)
- Формат A3 (основной), A4 (запасной)
- Без расчётов — только визуализация
- Стиль — чёрно-белый технический (не CAD)

## 9. Перспективы (после MVP)

### Этап 2: План расположения УУТЭ
- Зависит от выбранной принципиальной схемы (набор датчиков)
- Шаблон: вид сверху на трубопроводы с расположением датчиков и ШУ
- Клиент указывает: диаметр трубопровода, расстояние от стены
- Срок: ~3–5 дней

### Этап 3: План подключения объекта
- Самый простой шаблон: здание + теплосеть + граница балансовой принадлежности
- Клиент вводит: адрес, улицы, направление подключения
- Срок: ~2–3 дня

### Этап 4: Модульная сборка SVG
- Рефакторинг из 8 отдельных функций в модульный конструктор
- Позволит легко добавлять новые конфигурации без нового кода
- Срок: ~3–5 дней

### Этап 5: Расширение каталога
- Дополнительные конфигурации по запросам клиентов
- Одноступенчатые подогреватели
- Другие схемы включения вентиляции
