Общая схема пайплайна:
Клиент загружает ТУ (PDF) → LLM извлекает параметры → Агент определяет, чего не хватает → Отправляет клиенту письмо с запросом (план, схемы) → Клиент отвечает → Агент парсит ответ → Заполняет Excel → Excel подкладывается в T-FLEX → T-FLEX генерит чертежи + ПЗ → Экспорт PDF → Агент отправляет клиенту

Разбивка по модулям:
Модуль 1 — Парсинг ТУ (самый сложный)
ТУ приходят в двух вариантах: текстовый PDF и сканы. Для текстовых — извлечение текста через pymupdf, для сканов — OCR. Дальше текст отправляется в Claude/GPT с промптом, который знает структуру данных по 1036. LLM возвращает структурированный JSON с параметрами: тепловые нагрузки, параметры теплоносителя, диаметры, схема подключения и т.д.
Критичный момент — валидация. LLM может галлюцинировать цифры. Нужен слой проверки: диапазоны значений (давление не может быть 50 атм в жилом доме), перекрёстные проверки (нагрузка vs диаметр), обязательные поля по 1036. Всё что не прошло валидацию — помечается для ручной проверки.
Модуль 2 — Коммуникация с клиентом
Стейт-машина на каждую заявку. Состояния: new → tu_parsed → waiting_client_info → client_info_received → processing → completed. Письма по шаблонам через SMTP с вашего домена. В письме — конкретный список того, что нужно дослать, плюс образцы (PDF-приложения). Ответ клиента парсится: вложения сохраняются, текст анализируется LLM.
Модуль 3 — Заполнение Excel
Самый прямолинейный модуль. openpyxl пишет в нужные ячейки вашего шаблона Excel. Маппинг «параметр из JSON → ячейка в Excel» — конфиг-файл. После заполнения Excel сам пересчитывает формулы (нужен пересчёт через xlcalc или открытие в LibreOffice в headless-режиме, потому что openpyxl формулы не исполняет).
Модуль 4 — Автоматизация T-FLEX
Вот тут ключевой вопрос. T-FLEX CAD 17 имеет COM API и поддерживает макросы на C#. Сценарий: Python-скрипт через subprocess или COM-автоматизацию открывает T-FLEX, загружает параметрическую модель, подкладывает Excel, T-FLEX перестраивает модель, экспортирует PDF. Это работает на Windows-машине, где стоит T-FLEX с лицензией. Вариант — выделенный Windows-сервер или VM.
Модуль 5 — Оркестратор (сам агент)
Связывает всё вместе. Я бы рекомендовал не делать «один большой LLM-агент, который всё решает», а использовать LLM точечно — только там, где нужна «понималка»: парсинг ТУ, анализ ответа клиента, генерация текста писем. Остальное — детерминированный код.

Стек, который я бы предложил:
Python (FastAPI) как бэкенд, Celery + Redis для очередей задач (каждая заявка — цепочка задач), PostgreSQL для хранения заявок и их состояний, Claude API для парсинга и коммуникации, MinIO или S3 для файлов, Windows VM с T-FLEX для генерации чертежей.

Фазы разработки:
Первая фаза — парсинг ТУ. Это MVP-проверка всей идеи. Если LLM надёжно извлекает параметры из разных ТУ — всё остальное уже инженерия. Если нет — нужно думать о fallback (ручной ввод через форму).
Вторая фаза — заполнение Excel + генерация проекта в T-FLEX. Третья — полный email-флоу с клиентом.
